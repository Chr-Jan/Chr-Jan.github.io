Definitioner af Tekniske Termer

Term Definition

Azure:

 Microsoft Azure er en cloud-serviceplatform, der tilbyder et bredt udvalg af tjenester, herunder databehandling, lagring, netværk, kunstig intelligens og mere. Det giver virksomheder mulighed for at bygge, distribuere og administrere applikationer og tjenester globalt via Microsofts datacentre.

Azure-abonnement:

 Et Azure-abonnement giver brugere adgang til Azure-tjenester og fungerer som en enhed til styring, fakturering og skalerbarhed. Det giver adgang til at provisionere ressourcer inden for Azure-miljøet.

Azure-regioner:

 Azure-regioner er geografisk adskilte områder, hvor Microsofts datacentre er placeret. Disse regioner er spredt over hele verden, hvilket giver brugerne mulighed for at vælge et specifikt sted til at distribuere deres Azure-ressourcer.

Tilgængelighedszoner:

 Tilgængelighedszoner er fysisk adskilte datacentre inden for en Azure-region. Disse zoner er designet til at sikre høj tilgængelighed og fejltolerance for Azure-tjenester ved at sikre, at hvis en zone svigter, er andre stadig tilgængelige.

Azure-regionpar:

 Azure-regionpar er par af Azure-regioner, der er geografisk adskilte, men stadig tæt nok til at tillade replikation af data og tjenester mellem dem. Dette muliggør forretningskontinuitet og katastrofeberedskab ved at sikre, at data og tjenester replikeres til et andet område, hvis en region oplever nedetid.

Microsoft Entra ID:

 Microsoft Azure Active Directory (Azure AD) er en cloud-baseret mappe- og identitetsstyringstjeneste, der giver sikker adgangskontrol og identitetsbeskyttelse for Azure-tjenester, Microsoft 365 og andre cloud-applikationer.

Single Sign-On (SSO):

 Single Sign-On er en godkendelsesproces, hvor en bruger kun behøver at logge ind én gang for at få adgang til flere tjenester eller applikationer uden behov for gentagne loginsessioner.

Identitet:

 En identitet henviser til en enhed såsom en bruger, en applikation eller en enhed, der kan godkendes for at få adgang til ressourcer eller tjenester inden for et system.

Azure-tenant (Mappe):

 En Azure-tenant, også kendt som en mappe, er en dedikeret instans af Azure Active Directory, der repræsenterer en enkelt organisation. Hver tenant har sine egne brugere, grupper, politikker og sikkerhedsprincipper.

Azure-abonnement:

 Et Azure-abonnement er et betalingsforhold, der giver brugere adgang til Azure-tjenester og fungerer som en enhed til fakturering og ressourcestyring. Et abonnement er knyttet til en Azure-konto og giver adgang til at provisionere og administrere ressourcer inden for Azure-miljøet.

Microsoft Entra ID:

 Microsoft Entra ID er Microsofts skybaserede directory- og identitetsstyringstjeneste med flere lejere. Den hjælper med at understøtte brugeradgang til ressourcer og applikationer.
  1. Interne ressourcer og apps placeret på dit firmas netværk
  2. Eksterne ressourcer som Microsoft 365, Azure-portalen og SaaS-applikationer
  3. Cloud-applikationer udviklet til din organisation

Fremtrædende Funktioner:
  1. Single sign-on (SSO) adgang
  2. Omnidirektionel enhedssupport
  3. Sikker fjernadgang
  4. Cloud-udvidelighed
  5. Beskyttelse af følsomme data
  6. Understøttelse af selvbetjening

Ting at overveje ved brug af Microsoft Entra Funktioner:
  1. Aktivér SSO-adgang
  2. Byg en konsistent brugeroplevelse for alle enheder
  3. Beskyt lokale webapps med sikker fjernadgang
  4. Udvid til skyen for at administrere brugere konsistent
  5. Forbedr sikkerheden med identitetsbeskyttelsesfunktioner
  6. Udnyt selvbetjeningsfunktioner til omkostningsreduktion

Opret brugerkonti:

 Hver bruger, der ønsker adgang til Azure-ressourcer, har brug for en Azure-brugerkonto. En brugerkonto har alle de oplysninger, der kræves for at godkende brugeren under tilmeldingsprocessen. Microsoft Entra ID understøtter tre typer af brugerkonti. Typerne angiver, hvor brugeren er defineret (i skyen eller lokalt) og om brugeren er intern eller ekstern for din Microsoft Entra-organisation.

Ting at vide om brugerkonti:
  1. Cloud-identitet: En brugerkonto med en cloud-identitet er kun defineret i Microsoft Entra ID. Denne type brugerkonto inkluderer administratorbrugerkonti og brugere, der administreres som en del af din organisation. En cloud-identitet kan være for brugerkonti defineret i din Microsoft Entra-organisation og også for brugerkonti defineret i en ekstern Microsoft Entra-instans. Når en cloud-identitet fjernes fra den primære mappe, slettes brugerkontoen.
  2. Directory-synkroniseret identitet: Brugerkonti, der har en directory-synkroniseret identitet, er defineret i en lokal Active Directory. En synkroniseringsaktivitet finder sted via Microsoft Entra Connect for at bringe disse brugerkonti ind i Azure. Kilden til disse konti er Windows Server Active Directory.
  3. Gæstbruger: Gæstbrugerkonti er defineret uden for Azure. Eksempler inkluderer brugerkonti fra andre skyudbydere og Microsoft-konti som f.eks. en Xbox LIVE-konto. Kilden til gæstbrugerkonti er Inviteret bruger. Gæstbrugerkonti er nyttige, når eksterne leverandører eller entreprenører har brug for adgang til dine Azure-ressourcer.

Ting at overveje ved valg af brugerkonti:
  1. Overvej, hvor brugere er defineret. Afgør, hvor dine brugere er defineret. Er alle dine brugere defineret inden for din Microsoft Entra-organisation, eller er nogle brugere defineret i eksterne Microsoft Entra-instanser? Har du brugere, der er eksterne for din organisation? Det er almindeligt for virksomheder at understøtte to eller flere kontotyper i deres infrastruktur.
  2. Overvej support til eksterne bidragydere. Tillad eksterne bidragydere at få adgang til Azure-ressourcer i din organisation ved at understøtte brugerkontoen af typen Gæstbruger. Når den eksterne bidragyder ikke længere har brug for adgang, kan du fjerne brugerkontoen og deres adgangsprivilegier.
  3. Overvej en kombination af brugerkonti. Implementer brugerkontotyperne, der gør det muligt for din organisation at opfylde deres forretningskrav. Understøt brugerkonti med directory-synkroniseret identitet til brugere defineret i Windows Server Active Directory. Understøt cloud-identiteter til brugere defineret i din interne Microsoft Entra-struktur eller for brugere defineret i en ekstern Microsoft Entra-instans.

Administrer brugerkonti:

 Der er flere måder at tilføje brugerkonti til cloud-identitet i Microsoft Entra ID. En almindelig tilgang er ved at bruge Azure-portalen. Brugerkonti kan også tilføjes til Microsoft Entra ID gennem Microsoft 365 Admin Center, Microsoft Intune-administrationskonsollen og Azure CLI.

Detaljer om cloud-identitetskonti:

Hvordan cloud-identitetsbrugerkonti er defineret i Microsoft Entra ID: Administrator kan oprette en bruger inden for organisationen eller invitere en gæst til at give adgang til organisationsressourcer:

  1. Et nyt brugerkonto skal have et visningsnavn og et tilknyttet brugernavn. Et eksempel på et visningsnavn er Aran Sawyer-Miller, og det tilknyttede brugernavn kunne være asawmill@contoso.com.
  2. Information og indstillinger, der beskriver en bruger, er gemt i brugerkontoprofilen.
  3. Profilen kan have andre indstillinger som en brugers stillingsbetegnelse og deres kontaktemailadresse.
  4. En bruger med Global administrator eller Brugeradministrator privilegier kan foruddefinere profildata i brugerkonti, såsom hovedtelefonnummeret for virksomheden.
  5. Ikke-administratorbrugere kan angive nogle af deres egne profildata, men de kan ikke ændre deres visningsnavn eller kontonavn.

Ting at overveje ved håndtering af cloud-identitetskonti:
  1. Overvej brugerprofildata. Tillad brugere at angive deres profiloplysninger for deres konti, efter behov. Brugerprofildata, herunder brugerens billede, job og kontaktoplysninger, er valgfrie. Du kan også levere visse profilindstillinger for hver bruger baseret på din organisations behov.
  2. Overvej gendannelsesmuligheder for slettede konti. Inkluder gendannelsesscenarier i din kontohåndteringsplan. Gendannelsesoperationer for en slettet konto er tilgængelige op til 30 dage efter, at en konto er fjernet. Efter 30 dage kan en slettet brugerkonto ikke gendannes.
  3. Overvej indsamling af kontodata. Indsaml logindata og revisionslogoplysninger for brugerkonti. Microsoft Entra ID giver dig mulighed for at indsamle disse data for at hjælpe dig med at analysere og forbedre din infrastruktur.

Opret brugerkonti i bulk:

 Microsoft Entra ID understøtter flere bulk-operationer, herunder bulkoprettelse og -sletning af brugerkonti. Den mest almindelige tilgang til disse operationer er at bruge Azure-portalen. Azure PowerShell kan bruges til bulk-upload af brugerkonti.

Ting at vide om bulk-kontooperationer:
  1. Kun globale administratorer eller brugeradministratorer har privilegier til at oprette og slette brugerkonti i Azure-portalen.
  2. For at fuldføre bulkopret eller slette operationer udfylder administratoren en kommasepareret værdier (CSV) skabelon af dataene for brugerkontiene.
  3. Bulk-operationsskabeloner kan downloades fra Microsoft Entra-administrationscenteret.
  4. Bulklister over brugerkonti kan downloades.

Ting at overveje ved oprettelse af brugerkonti:
  1. Her er nogle designovervejelser for oprettelse og sletning af brugerkonti. Tænk over, hvilke brugerkontokonventioner og -processer der kan være påkrævet af din organisation.
  2. Etabler eller implementer en navngivningskonvention for dine brugerkonti. Anvend konventioner til brugerkontonavne, visningsnavne og brugeraliaser for ensartethed på tværs af organisationen. Konventioner for navne og aliaser kan forenkle bulkoprettelsesprocessen ved at reducere områder med unikhed i CSV-filen. En konvention for brugernavne kunne begynde med brugerens efternavn efterfulgt af et punktum og afslutte med brugerens fornavn, som f.eks. Sawyer-Miller.Aran@contoso.com.
  3. Implementer en konvention for det indledende kodeord for en nyoprettet bruger. Design et system til at meddele nye brugere om deres adgangskoder på en sikker måde. Du kan generere en tilfældig adgangskode og sende den pr. e-mail til den nye bruger eller deres leder.
  4. Se og håndter eventuelle fejl ved at downloade resultatfilen på siden Bulkoperationer i Azure-portalen. Resultatfilen indeholder årsagen til hver fejl. En fejl kan være en brugerkonto, der allerede er oprettet, eller en konto, der er duplikeret. Generelt er det nemmere at uploade og fejlfinde mindre grupper af brugerkonti.

Opret gruppekonti:

 Microsoft Entra ID giver din organisation mulighed for at definere to forskellige typer af gruppekonti. Sikkerhedsgrupper bruges til at administrere medlems- og computeradgang til fælles ressourcer for en gruppe brugere. Du kan oprette en sikkerhedsgruppe til en specifik sikkerhedspolitik og anvende de samme tilladelser til alle medlemmer af en gruppe. Microsoft 365-grupper giver samarbejdsmuligheder. Gruppemedlemmer har adgang til en fælles mailbox, kalender, filer, SharePoint-websted og mere.

Ting at vide om oprettelse af gruppekonti:
  1. Brug sikkerhedsgrupper til at angive tilladelser for alle gruppe medlemmer på samme tid, i stedet for at tilføje tilladelser til hver enkelt medlem individuelt.
  2. Tilføj Microsoft 365-grupper for at aktivere gruppeadgang for gæstbrugere uden for din Microsoft Entra organisation.
  3. Sikkerhedsgrupper kan kun implementeres af en Microsoft Entra-administrator.
  4. Normale brugere og Microsoft Entra-administratorer kan begge bruge Microsoft 365-grupper.

Ting at overveje ved tilføjelse af gruppe medlemmer:

 Når du tilføjer medlemmer til en gruppe, er der forskellige måder, du kan tildele medlemsadgangsrettigheder på. Når du gennemgår disse muligheder, skal du overveje, hvilke grupper der er nødvendige for at understøtte din organisation, og hvilke adgangsrettigheder der skal tildeles til gruppe medlemmer.

Adgangsrettigheder:
  1. Tildelt: Tilføj specifikke brugere som medlemmer af en gruppe, hvor hver bruger kan have unikke tilladelser.
  2. Dynamisk bruger: Brug dynamiske medlemskabsregler til automatisk at tilføje og fjerne gruppe medlemmer. Når medlemsattributter ændres, gennemgår Azure de dynamiske gruppe regler for kataloget. Hvis medlemsattributterne opfylder reglens krav, tilføjes medlemmet til gruppen. Hvis medlemsattributterne ikke længere opfylder reglens krav, fjernes medlemmet.
  3. Dynamisk enhed: (Kun sikkerhedsgrupper) Anvend dynamiske gruppe regler til automatisk at tilføje og fjerne enheder i sikkerhedsgrupper. Når enhedsattributter ændres, gennemgår Azure de dynamiske gruppe regler for kataloget. Hvis enhedsattributterne opfylder reglens krav, tilføjes enheden til sikkerhedsgruppen. Hvis enhedsattributterne ikke længere opfylder reglens krav, fjernes enheden.

Opret administrative enheder:

Når du designer din strategi for styring af identiteter og governance i Azure, er det afgørende at planlægge omhyggeligt for omfattende styring af din Microsoft Entra-infrastruktur. Det kan være nyttigt at begrænse det administrative område ved hjælp af administrative enheder for din organisation. Opdelingen af roller og ansvarsområder er især nyttig for organisationer, der har mange uafhængige divisioner.

Overvej følgende scenario:
  1. En stor universitet bestående af flere forskellige skoler som for eksempel Forretning, Ingeniørvidenskab og Medicin.
  2. Universitetet har administrative kontorer, undervisningsbygninger, sociale bygninger og studenterboliger.
  3. Hver forretningsafdeling har sit eget interne netværk til ressourcer som servere, printere og faxmaskiner.
  4. Hver undervisningsbygning er tilsluttet universitetets netværk, så både instruktører og studerende kan få adgang til deres konti.
  5. Netværket er også tilgængeligt for studerende og dekaner i studenterboliger og sociale bygninger.
  6. Gæstbrugere på universitetet har brug for adgang til internettet via universitetets netværk.

Ting at tænke på vedrørende administrative enheder:
  1. Skab en rolle, der har administrative tilladelser kun til Microsoft Entra-brugere i den administrative enhed for Ingeniørvidenskab.
  2. Opret en administrativ enhed for Ingeniørvidenskab.
  3. Populér den administrative enhed med kun Ingeniørvidenskabsstuderende, personale og ressourcer.
  4. Tilføj Ingeniørvidenskabsafdelingens IT-team til rollen sammen med dens omfang.

Ting at overveje ved arbejde med administrative enheder:
  1. Overvej ledelsesværktøjer. Gennemgå dine muligheder for at administrere AUs. Du kan bruge Azure-portalen, PowerShell-cmdlets og scripts eller Microsoft Graph.
  2. Overvej rollekrav i Azure-portalen. Planlæg din strategi for administrative enheder i henhold til rolleprivilegier. I Azure-portalen kan kun brugere med rollen Global Administrator eller Privileged Role Administrator administrere AUs.
  3. Overvej omfanget af administrative enheder. Forstå, at omfanget af en administrativ enhed kun gælder for administrationsrettigheder. Medlemmer og administratorer af en administrativ enhed kan udøve deres standardbrugerrettigheder til at gennemse andre brugere, grupper eller ressourcer uden for deres administrative enhed.

Azure Omkostninger og Faktorer

Faktor

Beskrivelse

Ressourcetype:

 Påvirker omkostninger; metered instances bruges til at beregne regningen.

Forbrug:

 Betal for det, du bruger; mere forbrug = højere omkostninger.

Vedligeholdelse:

 Hold øje med ressourcer for at kontrollere omkostninger.

Geografi:

 Omkostninger varierer afhængigt af region; global implementering = globale prisforskelle.

Netværkstrafik:

 Omkostninger baseret på zoner.

Abonnementstype:

 Visse abonnementer inkluderer brugsrettigheder, der påvirker omkostninger.

Azure Marketplace:

 Køb løsninger og tjenester fra tredjepartsleverandører; betal for både Azure-tjenester og tredjepartstjenester.

Pris- og Total Cost of Ownership-lommeregner

Prisberegneren:

  • Estimerer omkostninger ved ressourceallokering i Azure.
  • Tillader estimater for individuelle ressourcer eller fulde løsninger.
  • Fokus på omkostninger ved tildelte ressourcer.
  • Kun til informationsformål, ingen tildeling af ressourcer eller opkrævning af gebyrer.

TCO-lommeregner:

  • Sammenligner omkostninger ved on-premises og Azure-infrastrukturer.
  • Indtastning af nuværende infrastrukturkonfiguration.
  • Sammenligner forventede omkostninger mellem miljøer.
  • Indtastning af antagelser som strøm- og IT-arbejdsomkostninger.
  • Estimerer omkostningsforskellen for at køre samme miljø i datacenter eller Azure.

    Formålet med ressource-låse

    Ressource-låse forhindrer utilsigtede sletninger eller ændringer af ressourcer.

    Typer af ressource-låse

    • Slet: Forhindrer sletning af en ressource, men tillader læsning og ændringer.
    • Læs-kun: Tillader kun læsning af en ressource, forhindrer sletning og opdatering, ligner Reader-rollens tilladelser.

    Administrering af ressource-låse

    Ressource-låse kan administreres via Azure-portalen, PowerShell, Azure CLI eller Resource Manager-skabeloner.

    I Azure-portalen navigeres til Indstillinger for at tilføje, se, eller slette låse.

    Sletning eller ændring af låste ressourcer

    For at ændre en låst ressource, fjernes låsen først. Efterfølgende handlinger kræver stadig tilladelser, selvom brugeren ejer ressourcen.

    Formål med Service Trust Portalen

    • Portal for adgang til Microsofts sikkerheds-, privatlivs- og overholdelsesressourcer
    • Indeholder kontrol- og procesdetaljer for beskyttelse af skytjenester og kundedata
    • Kræver login med Microsoft Entra organisationskonto og accept af fortrolighedsaftale

    Adgang til Service Trust Portalen

    • Portalens webadresse: https://servicetrust.microsoft.com/
    • Hovedmenu med kategorier som "Service Trust Portal", "Min Bibliotek" og "Alle Dokumenter"
    • "Service Trust Portal": Hurtig adgang tilbage til portalens hjemmeside
    • "Min Bibliotek": Gem dokumenter til hurtig adgang og modtag opdateringsnotifikationer
    • "Alle Dokumenter": Central placering for alle portalens dokumenter, hvor brugere kan fastgøre dokumenter til personligt bibliotek

    Værktøjer til interaktion med Azure

    • Azure Portal: Webbaseret konsol til at administrere Azure-subscription med grafisk brugergrænseflade.
    • Azure PowerShell: Værktøj til at køre kommandoer (cmdlets) for at administrere Azure-ressourcer.
    • Azure CLI: Lignende Azure PowerShell, men bruger Bash-kommandoer.
    • Azure Cloud Shell: Browserbaseret shell til at administrere Azure-ressourcer med understøttelse af både Azure PowerShell og Azure CLI.

    Implementering af Azure-abonnementer

    Et Azure-abonnement er en logisk enhed af Azure-tjenester knyttet til en Azure-konto. Abonnementer hjælper med at organisere adgangen til cloudtjenester og styrer fakturering og betaling.

    • Hver Azure-tjeneste tilhører et abonnement.
    • Abonnementer har forskellige fakturerings- og betalingskonfigurationer.
    • Mere end én Azure-konto kan knyttes til et abonnement.
    • Fakturering sker per abonnement.
    • Brugere kan have flere abonnementer knyttet til deres Azure-konto.

    Overvej behovet for abonnementer, typer af Azure-konti, flere abonnementer og adgang til ressourcer for at effektivt implementere Azure-abonnementer.

    Få en Azure-abonnement

    For at bruge Azure skal du have en Azure-abonnement. Dette kan opnås gennem en Enterprise-aftale, en Microsoft-forhandler, en Microsoft-partner eller ved at åbne en personlig gratis konto til en prøveperiode.

    Identificer Azure abonnementsbrug

    Vi gennemgik måder, hvorpå du kan opnå et Azure-abonnement. Lad os nu se på de typer af Azure-abonnementer, der er tilgængelige.

    Azure tilbyder gratis og betalte abonnementsmuligheder

    Azure tilbyder gratis og betalte abonnementsmuligheder for at imødekomme forskellige behov og krav. De mest almindelige abonnementer er Gratis, Betal-som-du-går, Enterprise-aftale og Studerende. For din organisation kan du vælge en kombination af indkøbsmuligheder og abonnementsvalg for at imødekomme dine forretningsbehov.

    Overvejelser ved valg af Azure-abonnementer

    Når du tænker over, hvilke typer Azure-abonnementer der ville fungere for din organisation, skal du overveje følgende scenarier:

    • Azure gratis: Et Azure gratis abonnement inkluderer en monetær kredit til at bruge på enhver service i de første 30 dage. Du får gratis adgang til de mest populære Azure-produkter i 12 måneder og adgang til mere end 25 produkter, der altid er gratis. Et Azure gratis abonnement er en fremragende måde for nye brugere at komme i gang på.
    • Betal månedligt for brugte tjenester: Et Betal-som-du-går (PAYG) abonnement opkræver dig månedligt for de tjenester, du brugte i den pågældende faktureringsperiode. Denne abonnementstype er passende for et bredt spektrum af brugere, fra enkeltpersoner til små virksomheder og mange store organisationer.
    • Azure Enterprise-aftale: En Enterprise-aftale giver fleksibilitet til at købe skytjenester og softwarelicenser under én aftale. Aftalen kommer med rabatter til nye licenser og Software Assurance. Denne type abonnement henvender sig til organisations med virksomhedsskala.
    • Støtte til Azure for studerende: Et Azure for studerende abonnement inkluderer en monetær kredit, der kan bruges inden for de første 12 måneder.

    Studerende kan vælge gratis tjenester uden at angive et kreditkort under tilmeldingsprocessen. Du skal verificere din studenterstatus gennem din organisations e-mailadresse.

    Azure Storage-konti

    • En Azure Storage-konto giver et unikt navnerum til data, tilgængelig via HTTP/HTTPS fra overalt.
    • Data er sikre, tilgængelige, holdbare og skalerbare.
    • Lagringskontotypen bestemmer tjenester og redundansmuligheder.
    • Redundansmuligheder inkluderer LRS, GRS, RA-GRS, ZRS, GZRS, og RA-GZRS.
    • Forskellige kontotyper som Standard generel formål v2, Premium blokblobber, og Premium fildele dækker forskellige behov.
    • Unikke slutpunkter dannes af kontonavnet og Azure Storage-tjenesteslutpunktet.
    • Regler for navngivning inkluderer længdekrav og unikhed inden for Azure.

    Slutpunkter for Azure Storage-tjenester:

    • Blob Storage: https://.blob.core.windows.net
    • Data Lake Storage Gen2: https://.dfs.core.windows.net
    • Azure Files: https://.file.core.windows.net
    • Kø-lagring: https://.queue.core.windows.net
    • Tabel Storage: https://.table.core.windows.net

    Azure Storage Redundans

    • Azure Storage gemmer flere kopier af data for at beskytte mod fejl og katastrofer.
    • Der er to hovedtyper af redundans: i den primære region og i en sekundær region.
    • I den primære region tilbydes lokalt redundant storage (LRS) og zone-redundant storage (ZRS).
    • LRS replikerer data tre gange i samme datacenter, mens ZRS replikerer synkront på tværs af tre tilgængelighedszoner.
    • I en sekundær region kan data kopieres for at sikre høj holdbarhed.
    • Geo-redundant storage (GRS) og geo-zone-redundant storage (GZRS) er muligheder for redundans i sekundære regioner.
    • GRS replikerer data synkront og asynkront mellem regioner, mens GZRS tilbyder endnu højere holdbarhed og tilgængelighed.
    • Læsetilgang til sekundære regioner kræver aktivering af read-access geo-redundant storage (RA-GRS) eller read-access geo-zone-redundant storage (RA-GZRS).

    Azure Storage Services

    • Azure Blobs: Massivt objektlager til tekst og binære data, unstruktureret, tilgængeligt via HTTP/HTTPS.
    • Azure Files: Administrerede fildele i skyen via SMB/NFS protokoller, fuldt administreret, scripts og værktøjer til administration.
    • Azure Queues: Opbevaring af store beskedmængder, tilgængelig via HTTP/HTTPS, anvendes til asynkron bearbejdning.
    • Azure Disks: Blokniveau-lager til Azure VM'er, virtuelle diske administreret af Azure.
    • Azure Tables: Opbevaring af strukturerede data, NoSQL datastore, tilgængelig fra og udenfor Azure clouden.

    Azure Rollebaseret Adgangskontrol

    Zero Trust Model - Stikord Referat

    • Forventer indtrængen fra begyndelsen
    • Verificerer hver forespørgsel, som om den stammede fra et ukontrolleret netværk
    • Nødvendig på grund af moderne kompleksiteter og mobilitet i arbejdsstyrken
    • Microsoft anbefaler Zero Trust-model
    • Retningslinjer for Zero Trust-sikkerhedsmodel inkluderer:
      • Verificer eksplicit baseret på alle tilgængelige datapunkter
      • Begræns brugeradgang med JIT/JEA, risikobaserede politikker og datarbeskyttelse
      • Minimer blast radius, segmenter adgang, verificer end-to-end-kryptering
      • Brug analytics til synlighed, trusseldetektion og forbedrede forsvar
    • Tilpasning til Zero Trust involverer ændringer i traditionelle netværksstrukturer:
      • Ikke længere antage sikkerhed baseret på placering
      • Kræver godkendelse for alle enheder, uanset placering

    Opsætning af rollebaseret adgangskontrol

    • Sikker adgangsstyring til skyressourcer
    • Rollebaseret adgangskontrol (RBAC) hjælper med at styre, hvem der kan få adgang til Azure-ressourcer

    Ting at vide om Azure RBAC

    • RBAC giver detaljeret kontrol over, hvem der kan udføre hvilke handlinger på hvilke ressourcer
    • Eksempler på, hvad du kan gøre med Azure RBAC:
      • Tillad applikationer adgang til ressourcer i en ressourcegruppe
      • Lad brugere administrere specifikke ressourcetyper
      • Tillad grupper at administrere ressourcer på tværs af abonnementer

    Azure RBAC-koncepter

    • Sikkerhedsprincip: Brugere, grupper, serviceprincipper, forvaltede identiteter
    • Rolledefinitioner: Indbyggede eller brugerdefinerede sæt af tilladelser
    • Område: Rækkevidden af en rolletildeling, f.eks. abonnement, ressourcegruppe
    • Rolleudskrivning: Tildeling af rolledefinitioner til sikkerhedsprincipper på et bestemt område

    Overvejelser ved brug af Azure RBAC

    • Identificer anmodere: Brugere, applikationer og tjenester, der har brug for adgang
    • Definer roller baseret på jobansvar og arbejdsopgaver
    • Begræns omfanget af tilladelser for at sikre sikkerheden
    • Brug indbyggede eller brugerdefinerede rolledefinitioner efter behov

    Opret en rolledefinition

    • En rolledefinition består af sæt tilladelser defineret i en JSON-fil.
    • Hvert sæt tilladelser har et navn, f.eks. Handlinger (Actions) eller IkkeHandlinger (NotActions).
    • Eksempler på tilladelsessæt inkluderer Handlinger (Actions), IkkeHandlinger (NotActions), DataHandlinger (DataActions) og TildelbareOmråder (AssignableScopes).

    Ting at vide om rolledefinitioner

    • Azure RBAC tilbyder både indbyggede roller (built-in roles) og tilladelsessæt (permission sets) samt muligheden for at oprette brugerdefinerede (custom).
    • Ejeren (Owner) indbyggede rolle har højeste adgangsniveau i Azure.
    • Systemet bruger IkkeHandlingstilladelser (NotActions permissions) til at trække fra Handlingstilladelser (Actions permissions) for at bestemme effektive tilladelser.
    • TildelbareOmråder (AssignableScopes) angiver, hvor en rolledefinition kan tildeles, f.eks. administrationgrupper (management groups), abonnementer (subscriptions) osv.

    Rolle tilladelser

    • Brug Handlinger (Actions) og IkkeHandlinger (NotActions) sammen til at give eller nægte privilegier.
    Rolle (Role) Beskrivelse (Description) Handlingstilladelser (Actions permissions) IkkeHandlingstilladelser (NotActions permissions)
    Ejer (Owner) Tillad alle handlinger (Allow all actions) * n/a
    Bidragsyder (Contributor) Tillad alle handlinger undtagen skriv eller slet rolletildeling (Allow all actions except write or delete role assignment) *

    • Microsoft.Authorization/*/Skriv
    • Microsoft.Authorization/elevateAccess/Handling
    Læser (Reader) Tillad alle læsehandlinger (Allow all read actions) /*/læs n/a

    Rolleområder

    • Brug TildelbareOmråder (AssignableScopes) til at angive, hvor rollen kan tildeles.

    Overvejelser ved oprettelse af roller

    • Overvej at bruge indbyggede roller (built-in roles) eller opret egne brugerdefinerede roller (custom).
    • Tildel roller med det minimale nødvendige omfang for at udføre opgaver.
    • Identificer data eller ressourcer, der kun bør ændres under bestemte scenarier og anvend stram adgangskontrol.
    • Overvej at anvende afvisningsopgaver (deny assignments) for at blokere brugere fra specifikke handlinger, selvom en rolletildeling giver dem adgang.

    Opret en rolletildeling

    En rolletildeling er processen med at definere omfanget af en rolle for at begrænse tilladelser for en anmoder, såsom en bruger, gruppe, tjenestepal eller administreret identitet.

    Ting at vide om rolletildelinger

    • Formålet med en rolletildeling er at kontrollere adgangen.
    • Omfanget begrænser, hvilke tilladelser, der er defineret for en rolle, der er tilgængelige for den tildelte anmoder.
    • Adgangen fjernes ved at fjerne en rolletildeling.
    • En ressource arver rolletildelinger fra sin overordnede ressource.
    • De effektive tilladelser for en anmoder er en kombination af tilladelserne for de tildelte roller og tilladelserne for de roller, der er tildelt de anmodede ressourcer.

    Ting at overveje ved tildeling af omfangsniveauer for roller

    • Role Definition: Definition af, hvad hver rolle tillader brugeren at gøre.
    • Role Assignment: At tildele specifikke roller til brugere, grupper eller tjenestepaler.
    • Security Principal: Identifikation af den entitet, der anmoder om adgang og sikrer, at kun autoriserede personer får adgang.
    • Scope: Begrænsningen af, hvilke ressourcer eller data en rolle har adgang til.

    Sammenlign Azure-roller med Microsoft Entra-roller

    I Azure er der tre typer af roller til rådighed til adgangsstyring:

    • Klassiske abonnementsadministratorroller (Classic subscription administrator roles)
    • Azure rollebaserede adgangskontrol (RBAC) roller (Azure role-based access control (RBAC) roles)
    • Microsoft Entra administratorroller (Microsoft Entra administrator roles)

    For at forstå, hvordan disse forskellige roller er defineret og implementeret i Azure, er det nyttigt at kende noget af historien.

    • Da Azure først blev frigivet, blev adgang til ressourcer administreret med kun tre administratorroller: Kontoadministrator, Tjenesteadministrator og Medadministrator (Account Administrator, Service Administrator , and Co-Administrator). Adgang blev kontrolleret ved at tildele adminroller til abonnementer.
    • Senere blev rollebaseret adgangskontrol (RBAC) til Azure-ressourcer tilføjet. Azure RBAC er et nyere autorisationssystem, der giver finmasket adgangsstyring til Azure-ressourcer. RBAC inkluderer mange indbyggede roller, der kan tildeles på forskellige områder. Azure RBAC-modellen giver også mulighed for at oprette dine egne brugerdefinerede roller.
    • Ud over Azure RBAC-roller leverer Microsoft Entra ID indbyggede administratorroller til at administrere Microsoft Entra-ressourcer som brugere, grupper og domæner.
    Azure RBAC-roller (Azure RBAC roles) Microsoft Entra ID-administratorroller (Microsoft Entra ID admin roles)
    Adgangsstyring (Access management) Administrerer adgang til Azure-ressourcer (Manages access to Azure resources) Administrerer adgang til Microsoft Entra-ressourcer (Manages access to Microsoft Entra resources)
    Område tildeling (Scope assignment) Område kan specificeres på flere niveauer, herunder administrationsgrupper, abonnementer, ressourcegrupper og ressourcer (Scope can be specified at multiple levels, including management groups, subscriptions, resource groups, and resources) Område er specificeret på lejerniveau (Scope is specified at the tenant level)
    Rollederinitioner (Role definitions) Roller kan defineres via Azure-portalen, Azure CLI, Azure PowerShell, Azure Resource Manager-skabeloner og REST API (Roles can be defined via the Azure portal, the Azure CLI, Azure PowerShell, Azure Resource Manager templates, and the REST API) Roller kan defineres via Azure-administrationsportalen, Microsoft 365-administrationsportalen og Microsoft Graph PowerShell (Roles can be defined via the Azure admin portal, Microsoft 365 admin portal, and Microsoft Graph PowerShell)

    Implementering af rollebaseret adgangskontrol

    • Indbyggede roller til forskellige tjenester, opgaver og brugere
    • Mulighed for at tildele indbyggede roller på forskellige områder
    • Microsoft Entra ID tilbyder indbyggede roller til at administrere ressourcer såsom brugere, grupper og domæner
    • Tilgængelige administratorroller inkluderer Global admin, Application admin og Application developer
    • Microsoft Entra-administratorroller defineret på rodniveau for konfigurationen
    • Azure RBAC-roller giver mere detaljeret adgangsstyring til Azure-ressourcer
    • Disse roller kan anvendes på flere niveauer, inklusive roden, administrationsgrupper, abonnementer, ressourcegrupper eller ressourcer

    Gennemgang af grundlæggende Azure RBAC-roller

    • Azure tilbyder over 100 foruddefinerede rolledefinitioner.
    • Roller giver adgang til data inden for et objekt.
    • Eksempel: Hvis en bruger har læseadgang til data i en lagringskonto, kan de læse blobber eller beskeder i lagringskontoen.

    Grundlæggende rollebeskrivelser:

    • Owner: Har fuld adgang til alle ressourcer og kan delegere adgang til andre.
    • Contributor: Kan oprette og administrere alle typer af Azure-ressourcer, men kan ikke give adgang til andre.
    • Reader: Kan se eksisterende Azure-ressourcer.
    • User Access Administrator: Kan administrere brugeradgang til Azure-ressourcer.
